# RUNBOOK — эксплуатация edu-platform на Timeweb App Platform

Документ о том, **как правильно деплоить и обслуживать прод**, чтобы не повторять
аварии. Возник после серии падений из-за конфликта портов и удаления нужных доменов.

## Архитектура

| Компонент | App / ресурс | Домен / адрес | Ветка | Порт |
|---|---|---|---|---|
| Web (Next.js) | `edu-platform-web` (197446) | **https://губу.рф** | `tw-web` | 3000 |
| API (Fastify) | `edu-platform-api` (197482) | **https://api.губу.рф** | `tw-api` | 4000 |
| БД (Postgres) | `edu-platform-db` (4168202) | приватный `192.168.0.5` | — | 5432 |

- Все три — в **одной приватной сети** (`192.168.0.x`, network-2fb0a4871d1b41ccb40507ba44b34111).
- Поток запроса: браузер → **губу.рф** (web) → `/api-proxy` (серверная сторона Next.js) →
  `NEXT_PUBLIC_API_URL` = **https://api.губу.рф** → API → БД (приватно).
- Деплой: **push в `main` → GitHub Action `timeweb-deploy.yml`** синкает ветки
  `tw-api`/`tw-web` (main + `deploy/compose/*.yml` в корень) → Timeweb авто-собирает оба аппа.
- Файлы пользователей: пока **в PostgreSQL** (S3-бакет 507166 создан, но в код не подключён).

## ⛔ ГЛАВНОЕ ПРАВИЛО: один деплой за раз

Timeweb биндит **хост-порт** (`ports: "3000:3000"` / `"4000:4000"` — это штатный паттерн
Timeweb, **не убирать**). При деплое новый контейнер занимает порт, пока старый ещё держит его.
Если деплои **накладываются** (несколько подряд) или один падает/виснет — остаётся
**осиротевший контейнер на порту** → `Bind for 0.0.0.0:PORT failed: port is already allocated`
→ следующие деплои падают → пайплайн аппа **заклинивает** (действия отдают HTTP 500).

**Поэтому:**
- НЕ запускать новый деплой, пока идёт текущий. Ждать `ended_at`.
- НЕ делать серию правок подряд (env / тариф / домен / reboot) — **каждая = деплой**.
- Менять конфиг ОДНИМ изменением → дождаться завершения → проверить → только потом следующее.
- Доверять авто-деплою: один push в `main` = один деплой. Не дёргать руками параллельно.

> Именно нарушение этого правила (шторм быстрых деплоев) дважды заклинило API-апп
> и положило веб. Дисциплина деплоя — главная защита прода.

## Как правильно выкатывать изменения кода
1. Ветка → PR → merge в `main` (коммиты на русском, без `feat:`/`fix:`).
2. Merge запускает Action → синк `tw-api`+`tw-web` → авто-деплой.
3. Дождаться завершения **обоих** деплоев (не пушить следующее, пока идут).
4. Прогнать чек-лист проверки (внизу).

## Как менять переменные окружения
- Меняй на нужном аппе ОДНИМ сохранением → это один ребилд. Дождись `ended_at`.
- ⚠️ `NEXT_PUBLIC_API_URL` (web) — **BUILD-TIME** (вшивается при сборке Next.js). Значит
  это должен быть **публичный** адрес, доступный сборщику (`https://api.губу.рф`),
  а **НЕ приватный IP** — сборщик не в приватной сети, билд упадёт на сборе страниц.

## Домены — НЕ удалять нужные
- Web на `губу.рф`, API на `api.губу.рф` — **оба обязательны**.
- API обязан иметь **публичный** домен: и для сборки веба (Next.js дёргает API при `next build`),
  и чтобы фронт ходил в него. Приватного IP недостаточно.
- Технический `…twc1.net` можно удалять ТОЛЬКО если у аппа остаётся другой рабочий домен.
- Добавление домена к аппу — **только в панели** (App → Домены). DNS для домена,
  купленного у Timeweb, подхватывается автоматически; SSL (Let's Encrypt) — ~5–15 мин.

## Частые ошибки и лечение

### `Bind ... port is already allocated` (деплой падает на СТАРТЕ контейнера, образ собрался)
Осиротевший контейнер держит порт. Это НЕ про `CMD`/`EXPOSE` (они корректны).
1. НЕ запускать ещё деплои.
2. App → action **`pause`** (останавливает контейнеры, освобождает порт) → подождать →
   **`resume`** или ОДИН деплой при свободном порту.
3. Если `pause`/действия → 500 — апп заклинило (см. ниже).

### Апп заклинило (действия → `500 update_app_status_exception`, деплои висят/`stopping`)
Пайплайн аппа завис на стороне Timeweb.
1. Тикет в саппорт «почистите зависший контейнер/деплой», ИЛИ
2. Удалить апп и **пересоздать дисциплинированно** (чек-лист ниже).

### Почта (SMTP) не уходит
- Timeweb блокирует SMTP-порты **пер-апп** (анти-спам) → саппорт разблокирует по публичному
  IP аппа. **Повторять после каждого пересоздания** (новый IP снова заблокирован).
- Пароль приложения Яндекса — из **правильного** аккаунта (`bugrov.studio@yandex.ru`) +
  включён IMAP (Почта → Настройки → Почтовые программы). Иначе `535`.
- Тех-долг: `forgot-password` шлёт письмо синхронно → висит при недоступном SMTP (→ fire-and-forget).

### Приватная сеть
- Связь апп↔БД и апп↔апп задаётся **только в панели**. Пересоздание аппа теряет привязку →
  заводить заново (та же существующая сеть БД, свободный IP).

## Пересоздание аппа — дисциплинированный чек-лист
1. Создать апп: **Docker Compose**, репо через **GitHub-аккаунт** (авто-деплой), нужная ветка
   (`tw-api`/`tw-web`), регион **СПб**, **приватная сеть БД** (свободный IP), **БЕЗ переменных**.
2. Дождаться **полного** завершения деплоя создания (`ended_at`).
3. ОДНИМ изменением прописать все env:
   - API: `DATABASE_URL` (на `192.168.0.5`), `PORT=4000`, `HOST=0.0.0.0`, `NODE_ENV=production`,
     `JWT_SECRET`, `API_BASE_URL=https://api.губу.рф`, `CORS_ORIGIN=https://губу.рф`, `SMTP_*`.
   - Web: `NEXT_PUBLIC_API_URL=https://api.губу.рф`.
4. ОДИН деплой → дождаться.
5. Привязать домен в панели (App → Домены), если сменился.
6. Если пересоздан **API** → его публичный IP сменился → саппорт заново разблокирует SMTP-порты.
7. Между всеми шагами — ждать `ended_at`. Никаких параллельных правок.

## Чек-лист проверки после деплоя
- `https://api.губу.рф/health` → **200**
- `https://губу.рф` → **200**, вход (логин) → **200 + токен**
- Логи API: миграции применились, **нет `P1001`** (БД по приватной сети)
- Письмо: `forgot-password` → реальный приход в инбокс (не только ответ API)

## Где что лежит
- Прод-composes: `deploy/compose/api.yml`, `deploy/compose/web.yml` (в `main`). `ports:` НЕ трогать.
- Action: `.github/workflows/timeweb-deploy.yml`.
- Креды (на машине, не в репо): `~/.config/timeweb/` (token, db_password, jwt_secret),
  `~/.config/edu-platform/` (api_key, s3.env).
- Доска задач: Asana «edu-platform — бэклог».
