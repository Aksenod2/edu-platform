# Инженерная дисциплина команды (переносимое ядро)

> **Project-agnostic.** Это ядро применимо к ЛЮБОМУ проекту команды.
> Для нового проекта — копируется как есть; проектная специфика живёт в `CLAUDE.md`
> и памяти, а НЕ здесь. Выстрадано на первом проекте команды (OCS, прод-перенос
> 2026-06-04). Каждый агент и оркестратор обязаны это соблюдать.

## 0. Главный закон: «зелёно локально» ≠ «работает в проде»
Локальная среда почти всегда отличается от прод-формы (origin, моки, CDN, service worker,
tier, реальные куки/сессии). Целые классы багов видны ТОЛЬКО в прод-форме. Поэтому:
**локальные тесты — необходимое, но НЕ достаточное условие готовности.**

## 1. Гейты (обязательные проверки перед «готово»)
Применять по релевантности. Гейт = шаг, без которого задача не «done».

| Гейт | Когда | Чем |
|---|---|---|
| **Юнит/интеграция** | каждое изменение | тест-раннер проекта, зелёный baseline |
| **Code-review** | пачка изменений / рискованная логика | `/code-review` или агент-ревьюер — ловит баги спешки |
| **Security-review** | auth, секреты, доступ, деньги, raw-SQL, прод-конфиг | security-ревьюер, до мёржа |
| **🔴 Прод-проверка реальным браузером** | ЛЮБОЕ изменение, влияющее на прод | E2E реальным браузером против ПРОД-формы (не curl!) |

`curl`/юнит-тесты НЕ видят: cross-origin куки, service worker, CDN-кеш/rewrite, cold-start.
Прод проверяется только реальным браузером (Playwright/headless) против прод-URL.

## 2. Классы багов «расхождение сред» — чек-лист (смотреть ПЕРВЫМ при «не работает в проде»)
- **Cross-origin куки**: фронт и API на разных доменах → session-куки third-party → мобильные
  браузеры режут. Лечение: same-origin (прокси/один домен) → куки first-party.
- **Моки в проде**: dev-мок-воркер (MSW и т.п.) не выключен → перехватывает реальный API.
  Гейтить запуск воркера на прод-флаге.
- **Static rewrite / CDN порядок**: правило `/api/* → backend` должно быть ВЫШЕ SPA-fallback
  `/* → index.html`; CDN может отдавать HTML вместо JSON. Проверять content-type ответа.
- **Cold-start / пул соединений**: free-tier спит; после рестарта БД — протухшие коннекты →
  интермиттентные 5xx, пока пул не пересоздастся. Не путать с «багом кода».
- **Build-time env**: переменные фронта зашиваются в бандл при сборке — смена требует пересборки.

DevTools/Network — первый инструмент: Initiator (мок-воркер?), content-type (HTML?), status.

## 3. Источник правды
- **Функционал и роли — от документации, НЕ от грепа кода.** Каталог фич + матрица ролей =
  источник правды для тест-планов, спек, работы. Если доки разошлись с кодом — реконсилить,
  а не игнорировать.
- **Канон один, остальное — указатели.** При изменении реальности правится канонический
  документ; копии и промпты ссылаются на него, а не дублируют.
- **Прод-архитектура и «мины» — в памяти/хендоффе**, не теряются между сессиями.

## 4. Разделение знания
- **Трекер долгов/задач** (`docs/DEBTS.md` или внешний трекер — одно из двух, не оба) =
  что ДЕЛАТЬ. Источник работы.
- **Хендофф/память** = как УСТРОЕНО / «не сломай» (архитектура, мины, креды, доступы).
- Не дублировать бэклог в markdown и наоборот.

## 5. Оркестрация
- **Спека-цепочка ДО кода (обязательный гейт фичи).** Любая новая фича/волна начинается с
  product-analyst (юзер-кейсы, **ВСЕ точки входа**, план-матрица: сущность × операции ×
  состояния × роли × устройства) + ux-designer (потоки, размещение действий, полнота
  состояний), и лишь затем architect → код. НЕ срезать до «сразу архитектор ради скорости
  MVP»: точки входа и роли — обязательный пункт спеки, выносятся заказчику ДО реализации.
  Пропуск этого шага = зона ответственности оркестратора (урок: эпик встреч 1-на-1 ушёл с
  единственной точкой входа «карточка студента», пропустив очевидную «из расписания»).
- **Рамка аналитика — по СУЩНОСТИ, а не по запросу (системный анти-паттерн).** Когда
  работа касается сущности, аналитик описывает её ПОЛНЫЙ жизненный цикл (все операции ×
  обе роли × состояния) и помечает каждую операцию **сделано / частично / НЕ сделано**;
  список пробелов идёт заказчику ДО кода. Запрос заказчика — это ПОДМНОЖЕСТВО. Узкая рамка
  («проработай ровно то, что назвали») — корень повторяющихся дыр: пайплайн полон ровно
  настолько, насколько широка рамка от оркестратора. (Урок: встречи 1-на-1 — отдельными
  набегами всплывали «нельзя провести», «нет второй точки входа», «нет переноса».)
- **MVP-срез — только явной строкой и с согласием заказчика.** Любое «на старте не делаем X»
  фиксируется письменно (DEBTS) И выносится заказчику на «ок». Молча выпавшая из объёма
  операция жизненного цикла = мина, которую заказчик найдёт в проде. «MVP» — не лицензия
  тихо выкидывать CRUD/перенос/удаление.
- **Гейт полноты для НОВЫХ сущностей.** qa проверяет не только «паритет с легаси» (у новой
  сущности легаси нет — проверка пуста), но и полноту по матрице операций для ОБЕИХ ролей.
- **Оркестратор тестирует сам** (включая прод-браузер-гейт) и приносит заказчику РЕЗУЛЬТАТ,
  не баг на поиск. Заказчик — не QA.
- **Заказчика дёргать только под необратимо-ручное** (его аккаунты: облако, оплата, секреты,
  деплой-кнопки) — одной строкой, по-минимуму.
- **Автономия ≠ безответственность**: пушить проверенное без ожидания «ок», но safety-гейты
  (security-review, прод-браузер) остаются.
- **Батчить деплои**: каждый деплой = окно передеплоя/просадка прода. Не пушить часто подряд.
- **Прод-данные/DDL/секреты** — через гейт (руки заказчика или одобренный скрипт), не агентом молча.
- **Параллельные агенты — в изолированных worktree** с непересекающимися наборами файлов;
  общие «hot-zone» файлы правит оркестратор или строго один агент. Интеграция результатов —
  через git (merge), не через `cp` файлов: ворктри мог ответвиться от старой базы.

## 6. Меньше костылей — раньше думать об инфраструктуре
Если борьба со средой превращается в гонку костылей (warm-ping, ручные правила, обходы) —
поднять вопрос «правильной инфры» (платный tier, same-origin, staging) ЗАРАНЕЕ, а не после
серии провалов. Честно считать стоимость костыля против стоимости правильного решения.

## 6.5. Скиллы — какой и когда (playbook)
Скиллы делают агентов «умнее», если их доставать в нужный момент, а не делать руками.

| Скилл | Кто / когда |
|---|---|
| **`/code-review`** | Оркестратор/исполнитель — на пачку изменений перед пушем. Ловит баги спешки (на OCS нашёл 3 реальных). Гейт §1. |
| **`/security-review`** | security-reviewer — auth/секреты/деньги/изоляция данных/raw-SQL/прод-конфиг, до мёржа. |
| **`/verify`, `/run`** | release-qa, исполнители — РЕАЛЬНО запустить приложение и подтвердить поведение, а не «по коду должно работать». |
| **`/simplify`** | исполнитель — после фичи: чистка дублей/сложности (без охоты на баги). |
| **`deep-research`** | devops/tech-lead/system-analyst — при НЕИЗВЕСТНОМ поведении платформы/тех. Ресёрч ВМЕСТО проб-в-проде — на OCS сэкономил бы день. |
| **`skill-creator`** | завести кастомные командные скиллы (напр. `/prod-verify` — обёртка release-qa). |
| **`/fewer-permission-prompts`** | оркестратор периодически — снизить трение пермишенов. |
| **`/schedule`, `/loop`** | мониторинг/повторяющееся (прод-health, периодический смоук). |

Правило: упёрся в незнание платформы → `deep-research`, не методом тыка в проде. Заявляешь
«работает» → подтверди `/verify`/release-qa. Перед пушем рискового → `/code-review`.

## 7. Интерфейс стейкхолдера и уровни доверия

**Принцип:** чем меньше стейкхолдер видит и понимает область, тем ВЫШЕ требования к
прозрачности отчётов из неё. Видимую область (например, UI) он контролирует сам —
глазами; невидимая (бэкенд, инфра, данные) без принудительной прозрачности становится
чёрным ящиком, где копятся тихие долги. Выстрадано: на OCS бэк был чёрным ящиком для
стейкхолдера — именно там скопились тихие долги, всплывшие только на ретроспективе.

**Механика прозрачности:**
- **Журнал работ простым языком** — файл в репо (не чат): каждый блок работ из
  «невидимой» области завершается записью «что изменилось / как проверили / что это
  значит для стейкхолдера». Свежие сверху; стейкхолдер читает по запросу.
- **Доказательства поведением, не артефактами процесса**: «можно увидеть вот это»,
  а не «тесты зелёные» / «миграция применена». Закон: работа без понятного
  стейкхолдеру доказательства поведения = не принята.
- **Перевод развилок в продуктовый язык**: технический выбор доносится как варианты
  с продуктовыми последствиями («Вариант А: быстрее, но...; Вариант Б: ...»).
  Развилки без продуктовых последствий команда решает сама и фиксирует почему.
- **Запрет жаргона наверх**: в отчётах стейкхолдеру — только поведение и продукт;
  технический термин допустим лишь с переводом.

**Уровни доверия** (контроль ослабляется осознанно, не сам собой):
- **СТРОГИЙ** — новая команда / новый проект: каждый блок работ → запись в журнал +
  доказательство поведением; все развилки наверх в продуктовом виде.
- **СТАНДАРТНЫЙ** — после 1–2 чистых проектов: журнал ведётся, стейкхолдера дёргаем
  только по развилкам с продуктовыми последствиями.
- **ЛЁГКИЙ** — зрелая команда с накопленным доверием: журнал по вехам (релизы,
  крупные блоки), остальное — внутри команды.

Переход между уровнями — ТОЛЬКО явным решением стейкхолдера, не дрейфом
(«давно не писали в журнал — значит, можно не писать» — это деградация, не доверие).

## 8. Как переносить команду на новый проект
1. Скопировать каркасы ролей и перенацелить на стек/пути нового проекта: заполнить
   `<project-specifics>` в каждой (роли те же: pm, tech-lead, frontend, backend,
   security-reviewer, qa, analysts, devops, design-lead, **+ release/prod-QA**).
2. Скопировать ЭТО ядро (`engineering-discipline.md`) как есть — оно проектонезависимо.
3. Новый `CLAUDE.md` — проектная специфика (стек, порты, секреты, прод-устройство).
4. Завести `docs/DEBTS.md` + журнал работ + хуки-предохранители.
5. Первым делом — поднять прод-браузер-гейт (release/prod-QA агент), а не оставлять «на потом».
6. Прогнать экзамены команды (`evals/EXAMS.md` из шаблона) — убедиться, что правила
   работают, а не просто скопированы.
