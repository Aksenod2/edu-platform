# Проработка фичи (спека до кода)

Дёргать, когда заказчик просит новую фичу или смену поведения (не косметику/мелкий фикс).
Цель — увидеть ВЕСЬ объём и дыры ДО кода, чтобы заказчик не находил их в проде.

## Шаги
1. **Рамка — по СУЩНОСТИ, а не по узкому запросу.** Запрос заказчика — подмножество.
   Берём затронутую сущность (Встреча, Урок, Поток, Студент…) и её полный жизненный цикл.
2. **`product-analyst`** строит по своему методу (`.claude/agents/product-analyst.md`):
   карту историй по ролям + матрицу операций (создать/список/деталь/редактировать/
   **перенести**/**удалить**/архив + переходы статуса) × роли × состояния + аудит кода
   ✅/🟡/❌ с пруфами + **Example Mapping** (красные карточки-вопросы вскрывают забытые ветки).
3. **`ux-designer`** — потоки, размещение действий, полнота состояний, ВСЕ точки входа.
4. **MVP-срез — явной строкой и с согласием заказчика.** «НЕ делаем сейчас: X, Y» → DEBTS +
   наверх. Молча выпавшая операция = мина.
5. **`architect`** — реализуемость, миграции, разбивка. Только после спеки.
6. Объём согласуем с заказчиком ДО кода.

## Гейт (нельзя пропустить)
- Без матрицы покрытия (с пометками сделано/частично/нет) и без явного MVP-среза — в код НЕ идём.
- Связано: `docs/product/vision.md` (фильтр ценности), канон `.claude/team/engineering-discipline.md` §5.
