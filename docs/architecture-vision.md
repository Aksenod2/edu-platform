# Архитектура: комбинаторная модель контента

> Живой документ. Здесь фиксируем продуктовые решения по архитектуре платформы,
> чтобы они не терялись между сессиями. Дополняется по мере обсуждения тем.
> Статус: в работе (обсуждаем тему «базовая структура»).

## Главный принцип — комбинаторика

Контент модульный. Преподаватель постоянно создаёт **блоки** (уроки) по мере того,
как сам узнаёт новое, — база растёт и идёт чуть впереди программ. Из этих блоков
**комбинируются** разные форматы: один блок → менторская, несколько → интенсив,
большая часть → курс.

Ключевая идея: **урок не принадлежит одному курсу.** Это переиспользуемый кирпич,
который входит в разные конструкции. Контент пишется один раз — и «перепродаётся»
сколько угодно раз через новые запуски.

Цитата заказчика, задающая смысл:
«я хочу чтобы это падало в копилку и я мог это перепродать ещё 10, 50, 100, тысячу раз».

## Три слоя

### 1. Урок (блок) — атом контента
- Видео, материалы, тезисы, шаблон задания.
- **Ничей**: не привязан к потоку. Лежит в общей копилке.
- Это растущая база знаний преподавателя.

### 2. Сборка (программа) — комбинация блоков
- Упорядоченный набор уроков + тип: **курс / интенсив / менторская**.
- Это «продукт», который продаётся и переиспользуется.
- Один урок входит в несколько сборок (связь многие-ко-многим, с порядком).

### 3. Поток — запуск сборки для группы
- Конкретный набор людей + чат + расписание.
- «Перепродать 100 раз» = 100 потоков одной сборки.
- Сборка пишется один раз, потоков под неё — сколько угодно.

### Под капотом: Занятие (сессия)
- Проведение конкретного урока внутри конкретного потока: дата, время, ссылка на
  созвон, статус, запись.
- Нужно потому, что один урок в разных потоках идёт в разные даты и со своей записью.
- Для преподавателя это просто «урок в расписании потока».

## Скелет сущностей (целевой)

```
Урок (Lesson)        контент, БЕЗ привязки к потоку (убираем streamId)
Сборка (Program)     name, type: курс | интенсив | менторская
СборкаУрок           program <-> lesson + порядок (многие-ко-многим)
Поток (Stream)       name, -> Сборка, статус, группа, чат
Занятие (Session)    поток x урок: дата, время, ссылка, статус, запись
Запись студента      студент x задание в рамках потока
```

Отличие от текущей схемы: сейчас `Lesson.streamId` и `Assignment.streamId`
обязательны — контент одноразовый. В целевой модели уроки независимы, а даты
и записи переезжают на уровень «Занятия» (поток × урок).

## Как ложатся форматы заказчика

- **Большой курс** (Денис + Серёжа) — одна Сборка(курс); наборы студентов = потоки.
  Между наборами правится сама сборка/уроки.
- **Курс Серёжи** (дизайн) — отдельная Сборка(курс) со своими потоками. Независим.
- **Менторские** — Сборка(менторская) из нужных блоков; поток с регулярными занятиями.
  Часто переиспользует фрагменты большого курса (те же блоки).
- **Интенсив** — Сборка(интенсив) из нескольких блоков.

Всё собирается из одной общей копилки уроков.

## Уже принятые решения

- **Задания.** Одно задание на урок, раздаётся **всем** ученикам потока автоматически,
  каждый делает свою копию. Ручного назначения конкретному ученику нет — убираем.
- **Материалы.** Живут на уроках. Пункт меню «Материалы» — агрегатор: быстрый доступ
  ко всем материалам сразу за всё обучение. Отдельную «Библиотеку» не вводим.
  (Открытый нюанс: материалы «вообще по курсу», не привязанные к уроку — решить позже.)

## Открытые темы (повестка, обсуждаем по одной)

1. Базовая структура (Урок / Сборка / Поток) — **согласовано, фиксируется этим документом**.
2. Несколько преподавателей — кто чьих студентов видит, нужны ли роли.
3. Студенты — запись, онбординг, масштаб.
4. Расписание и менторские — повторяющиеся встречи, «день/неделя» преподавателя.
5. Материалы — добить нюанс с «общими по курсу».
6. Задания — добить остаток (механика согласована).
7. Проверка и фидбэк — каким должен быть разбор работы.
8. Прогресс студента — как видеть, где кто находится.
9. Автоматизация: API + Claude проверяет домашки (в самом конце).
