## Продуктовый подход (ОБЯЗАТЕЛЬНО — с 2026-06-18)

- **Продуктовый якорь:** `docs/product/vision.md` — видение + главная метрика-маяк
  (**доля студентов, доводящих задания/курс до результата**; НЕ выручка). Утверждён
  заказчиком. Каждая фича сверяется: двигает ли маяк?
- **Фильтр на входе каждой задачи (4 вопроса):** какую проблему решаем · для кого
  (студент/преподаватель) · какой результат ждём · как поймём, что помогло. Нет ответов —
  задачу не берём. Не «фабрика фич»: строим проблемы/исходы, а не список фич.
- **Продуктовый мандат — у оркестратора** (видение, маяк, приоритеты). Приоритизация —
  ICE (важность × уверенность × лёгкость) + MoSCoW-разрез волны. Новых продукт-ролей НЕ
  заводим — мандат на оркестраторе, фильтр ценности — у `product-analyst` на входе.
- **Системное улучшение процесса — эпик #174** (замки выпуска, общий контракт Zod,
  продуктовый слой). Принцип: механизмы-замки, а не правила-пожелания. Ретро —
  `docs/retro/2026-06-18.md`.

## Инженерная дисциплина и гейты (ОБЯЗАТЕЛЬНО — с 2026-06-18)

- **Канон команды:** `.claude/team/engineering-discipline.md` — обязателен к соблюдению
  всеми агентами и оркестратором. Здесь — выжимка, канон не дублируем.
- **Главный закон: «зелёно локально» ≠ «работает в проде».** Локальные тесты/сборка
  необходимы, но НЕ доказывают прод: классы багов (build-time `NEXT_PUBLIC_*` env,
  cross-origin куки, service worker, CDN/rewrite, cold-start, ТСПУ-блокировки исходящих)
  видны только в прод-форме. См. чек-лист расхождения сред в каноне §2.
- **🔴 Прод-проверка реальным браузером — обязательный гейт** перед показом заказчику
  или статусом «готово». Делает роль `release-qa` (Playwright против прод-URL `губу.рф`),
  не `curl`. **Заказчик — НЕ QA:** приносим РЕЗУЛЬТАТ, а не баг на поиск.
- **Гейт сборки перед пушем:** перед каждым push в `main` — полная прод-сборка
  `pnpm turbo build` (строже `tsc --noEmit`). Автоматизировано хуком
  `.claude/hooks/pre-push-build-gate.sh` (блокирует push без свежей сборки; обход —
  `EDU_SKIP_BUILD_GATE=1`, только осознанно).
- **Спека → реализация:** новая фича / смена поведения — сначала согласованная спека
  (`product-analyst` → `architect`), потом код. Косметика/мелкий фикс — сразу.
- **DEBTS:** любой хак / отложенный фикс / мина / красный тест — строка в `docs/DEBTS.md`.
  Долг без строки «не существует» и всплывёт в худший момент.
- **Журнал работ:** изменения в «невидимых» заказчику областях (бэкенд, инфра, данные)
  фиксируем простым языком в `docs/journal/<область>.md` (что изменилось / как проверили
  ПОВЕДЕНИЕМ / что значит заказчику).
- **Уровень доверия стейкхолдера: СТРОГИЙ.** Каждый блок «невидимой» работы → запись в
  журнал + доказательство поведением; развилки наверх в продуктовом виде. Смена уровня —
  только явным решением заказчика (см. канон §7).
- **Батчить деплои:** каждый push в `main` = окно передеплоя прода; не пушить часто подряд.

## Прод и деплой (ВАЖНО — актуально с 2026-05-22)

- **Прод работает на VPS** `edu-platform-vps` (Timeweb VPS, id 8030462, IP 37.77.105.70),
  каталог `/root/edu-platform/`. Стек: Docker Compose (`docker-compose.vps.yml`) + Caddy
  (reverse-proxy, TLS, домены губу.рф / 0снова.рф). Переменные окружения — в `.env.vps`
  НА СЕРВЕРЕ (в репозиторий не коммитятся: там ключи S3/SMTP и пр.).
- **Доставка кода в прод — АВТОМАТИЧЕСКАЯ из `main`.** Прод тянется сам: GitHub Action
  `.github/workflows/vps-deploy.yml` на каждый push в `main` (и по `workflow_dispatch`)
  заходит на VPS по SSH (секреты `VPS_SSH_*`) и выполняет `git fetch origin main` +
  `git reset --hard origin/main` + `docker compose --env-file .env.vps -f
  docker-compose.vps.yml up -d --build` + `docker image prune -f`. **Значит мерж/пуш в
  `main` сам выкатывает изменения на прод** — отдельно заходить на VPS не нужно. Следить
  за статусом выката — во вкладке Actions (workflow «Deploy to VPS»). Ручной запуск —
  тем же `workflow_dispatch`. Из облачной (web) сессии прямого SSH к VPS нет, но это и не
  требуется — деплоит CI.
- **Timeweb App Platform СНЕСЁН.** Поэтому это МЁРТВОЕ ЛЕГАСИ — не использовать и не верить:
  GitHub Action `timeweb-deploy.yml`, ветки `tw-api`/`tw-web`, `deploy/compose/*`,
  `deploy/TIMEWEB.md`, домен `api.губу.рф`, любые «авто-деплой по push → App Platform».
- **Файлы — S3 (live).** Файлы пишутся в Timeweb Object Storage (env-driven `lib/s3.ts`);
  `S3_*` заданы в `.env.vps` на VPS. На чтении пока сохранён фолбэк в PostgreSQL для старых
  файлов — **не убирать**, пока не сделана миграция `fileStorage → S3`.
- **Известный баг:** публичный `/api-proxy` (Next.js rewrite) исторически не пробрасывал
  заголовок `Range` → видео отдавалось целиком (200), а не 206 (перемотка/Safari). Чинится
  в прокси-роуте или отдачей `/files` напрямую через Caddy.

## Метод работы: декомпозиция и качество

Чтобы не терять базовые вещи (как однажды потеряли «удаление урока»), работаем так:

- **Метод аналитика — User Story Mapping + матрица покрытия + Example Mapping** (см.
  `.claude/agents/product-analyst.md`). Рамка — по СУЩНОСТИ, а не по узкому запросу:
  для затронутой сущности — карта историй по ролям + матрица операций (создать / список /
  деталь / редактировать / **перенести** / **удалить** / архив + спец-переходы статуса) ×
  роли × состояния, с пометкой **сделано / частично / нет** (аудит кода с пруфами).
  Example Mapping (красные карточки-вопросы) вскрывает забытые ветки ДО кода. Запрос
  заказчика — подмножество; полную карту согласуем с заказчиком до кода.
- **MVP-срез — только явной строкой и с согласием заказчика** (DEBTS + наверх). Молча
  выпавшая операция жизненного цикла = мина.
- **Definition of Done на каждую карточку** (проверяет `qa-engineer`): полный CRUD,
  все состояния, **паритет с легаси** (новый экран не растерял функции старого —
  сверяем grep'ом), гейты зелёные, дан чек-лист ручной проверки на проде.
- **Паритет с легаси обязателен** при замене старого экрана новым.
- **Ревью перед мержем:** `code-reviewer` (и `security-reviewer` для чувствительных
  мест) читают дифф.
- **Аудит после волны:** `qa-engineer` проверяет полноту CRUD/состояний по всем новым
  экранам и заводит карточки на пробелы.
- **Декомпозируй щедро:** лучше больше мелких задач по ролям, чем одна крупная — так
  качественнее и параллельнее.

  ```bash
  npx shadcn add <component> -c apps/web
  ```
